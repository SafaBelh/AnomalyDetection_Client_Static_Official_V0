# Ask&Go AnomalyIQ Widgets Demo

This project demonstrates how an ERP application can integrate AnomalyIQ widgets as plug-and-play web components. It contains:

- `backend/`: a mock Express API that simulates the ERP/API side.
- `frontend/`: a React/Vite SaaS configurator plus the `/askgo` ERP demo page.
- Web components registered by the frontend, such as `anomaly-alerts`, `anomaly-score`, `anomaly-budget`, and `anomaly-forecast`.

The important integration flow is: configure widgets in the SaaS app, generate the integration guide, then mount the web components in the ERP with a short-lived SSO token and widget API key.

## Quick Start

Install dependencies:

```bash
npm run install:all
```

Start the mock backend:

```bash
npm run dev:backend
```

Start the frontend:

```bash
npm run dev:frontend
```

Open:

- SaaS/configurator: `http://localhost:5173`
- Ask&Go ERP demo: `http://localhost:5173/askgo`

The frontend proxies `/api/*` to `http://localhost:3001/*` through `frontend/vite.config.ts`.

## Frontend Pages

### SaaS Configurator

The SaaS app lets you:

- Manage ERP entries.
- Add Ask&Go widgets to the Ask&Go client.
- Configure widget status, colors, border radius, fonts, API keys, and widget-specific settings.
- Generate an ERP integration report.
- View examples for Angular, Vue, and legacy Java Servlet/JSP integration.

Ask&Go style fields are shared across Ask&Go widgets. When you change style for any Ask&Go widget, the shared visual fields are propagated to the other Ask&Go widgets and reflected on `/askgo`.

### `/askgo`

The `/askgo` page simulates the ERP that consumes the configured widgets. It reads the same local SaaS widget configuration and updates when the SaaS config changes.

It mounts the widgets as web components:

- `anomaly-alerts`
- `anomaly-alert-detail`
- `anomaly-score`
- `anomaly-budget`
- `anomaly-forecast`

## Integration Steps

### 1. Include The Widget Bundle

Add the AnomalyIQ script once in the ERP shell, usually in `<head>` or before `</body>`:

```html
<script type="module" src="https://cdn.anomalyiq.com/widgets.esm.js"></script>
```

For local demo testing, the React app registers the web components directly through `registerAnomalyWidgets()`.

### 2. Exchange ERP Session For AnomalyIQ Token

Your ERP backend must exchange the ERP user session for a short-lived AnomalyIQ token.

Example:

```js
const response = await fetch('https://api.anomalyiq.com/auth/sso-exchange', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json',
    'X-Api-Key': '<WIDGET_API_KEY>',
  },
  body: JSON.stringify({ erpToken: currentUser.sessionToken }),
});

const { token } = await response.json();
```

In this local demo, the mock endpoint is:

```http
POST /api/sso/exchange
```

It returns:

```json
{ "token": "fake-jwt-token-abc123" }
```

### 3. Mount The Web Component

Example for alerts:

```html
<anomaly-alerts
  token="<SSO_TOKEN>"
  api-key="<WIDGET_API_KEY>"
  erp-id="client_askgo_whitecape"
  endpoint="https://api.anomalyiq.com"
  status-filter="PENDING"
  refresh-key="0"
></anomaly-alerts>
```

Required common attributes:

| Attribute | Required | Description |
| --- | --- | --- |
| `token` | Yes | Short-lived SSO token generated by the ERP backend. |
| `api-key` | Yes | Widget API key generated by the SaaS. |
| `erp-id` | Yes | ERP/client ID. |
| `endpoint` | No | API base URL. Defaults to the production API if omitted. |

### 4. Pass JS Properties When Needed

Some widgets require JavaScript properties, not only HTML attributes.

Alert detail example:

```js
const detail = document.querySelector('anomaly-alert-detail');
detail.alert = selectedAlert;
```

Invoice score example:

```js
const score = document.querySelector('anomaly-score');
score.invoiceData = {
  supplier: 'Fournisseur SA',
  amount: 12450,
  date: '2024-06-15',
};
```

### 5. Listen To Custom Events

Widgets communicate with the ERP using DOM `CustomEvent` events.

Example:

```js
document.querySelector('anomaly-alerts').addEventListener('anomaly-select-alert', event => {
  const alert = event.detail;
  console.log('Selected alert', alert);
});
```

## Authentication

The intended production auth model uses two credentials:

- `Authorization: Bearer <sso_token>` identifies the current ERP user session.
- `X-Api-Key: <widget_api_key>` identifies the widget and ERP configuration.

The token should be short-lived, for example 15 minutes. The ERP backend should refresh it before expiration. Do not expose long-lived secrets in client-side code.

In this demo:

- `POST /api/sso/exchange` returns a fake token.
- The frontend passes `Authorization: Bearer fake-jwt-token-abc123` for widget data requests.
- The mock backend does not validate the token. It is only demonstrating the integration pattern.

## API Endpoints

The local frontend calls `/api/...`; Vite rewrites this to the backend without the `/api` prefix.

### SSO Exchange

```http
POST /sso/exchange
```

Returns a short-lived token.

Response:

```json
{ "token": "fake-jwt-token-abc123" }
```

### Alerts List

```http
GET /alerts?status=PENDING&supplier=Acme&page=0&size=10
```

Query params:

| Param | Type | Required | Description |
| --- | --- | --- | --- |
| `status` | string | No | `PENDING`, `CONFIRMED`, `REJECTED`, or `IGNORED`. |
| `supplier` | string | No | Case-insensitive supplier filter. |
| `page` | number | No | Zero-based page number. Default: `0`. |
| `size` | number | No | Page size. Default: `10`. |

Response contains `items`, `total`, `page`, `size`, and `totalPages`.

### Alert Detail

```http
GET /alerts/:id
```

Path params:

| Param | Type | Required | Description |
| --- | --- | --- | --- |
| `id` | string | Yes | Alert ID, for example `alert_001`. |

Returns one alert with explanation and reference values.

### Feedback

```http
POST /feedback/:alertId
```

Path params:

| Param | Type | Required | Description |
| --- | --- | --- | --- |
| `alertId` | string | Yes | Alert ID to update. |

Body:

```json
{
  "action": "CONFIRM",
  "comment": "Reviewed by AP team"
}
```

Body params:

| Param | Type | Required | Description |
| --- | --- | --- | --- |
| `action` | string | Yes | `CONFIRM`, `REJECT`, or `IGNORE`. |
| `comment` | string | No | Optional reviewer comment. |

### Invoice Score

```http
POST /pipelines/:id/invoices/check
```

Path params:

| Param | Type | Required | Description |
| --- | --- | --- | --- |
| `id` | string | Yes | Pipeline ID, for example `pipeline_askgo`. |

Body:

```json
{
  "supplier": "Acme Corp",
  "amount": 14500,
  "date": "2024-06-15",
  "label": "Consulting services",
  "extraFields": {}
}
```

Body params:

| Param | Type | Required | Description |
| --- | --- | --- | --- |
| `supplier` | string | Yes | Supplier name. |
| `amount` | number | Yes | Invoice amount. |
| `date` | string | Recommended | Invoice date in `YYYY-MM-DD` format. |
| `label` | string | No | Invoice label or service. |
| `extraFields` | object | No | Additional ERP-specific fields. |

Response:

```json
{
  "score": 82,
  "severity": "HIGH",
  "explanation": "High anomaly score..."
}
```

### Confirm Invoice

```http
POST /pipelines/:id/invoices/confirm
```

Body params:

| Param | Type | Required | Description |
| --- | --- | --- | --- |
| `supplier` | string | Yes | Supplier name. |
| `amount` | number | Yes | Invoice amount. |
| `date` | string | Yes | Invoice date. |
| `label` | string | No | Invoice label. |
| `extraFields` | object | No | Additional ERP fields. |

### Budget

```http
GET /pipelines/:pipelineId/series/:seriesId/budget?year=2024
```

Path/query params:

| Param | Type | Required | Description |
| --- | --- | --- | --- |
| `pipelineId` | string | Yes | Budget pipeline ID. |
| `seriesId` | string | Yes | Budget series ID. |
| `year` | number | No | Fiscal year. Default: `2024` in the mock backend. |

Returns annual variance and month-by-month actual/expected values.

### Forecast


Path/query params:

| Param | Type | Required | Description |

| `page` | number | No | Zero-based page number. Default: `0`. |

### Procurement Helpers

These endpoints support the Ask&Go mock ERP screens:

| Method | Path | Description |
| --- | --- | --- |
| `GET` | `/suppliers?status=ACTIVE&page=0&size=20` | Supplier list. |
| `GET` | `/suppliers/:id` | Supplier detail. |
| `POST` | `/suppliers/:id/status` | Update supplier status. |
| `GET` | `/departments` | Department list. |
| `GET` | `/invoices?supplier=Acme&page=0&size=10` | Saved invoice history. |
| `POST` | `/reset` | Reset mock in-memory data. |

## Widget Events

| Event | Emitted By | Payload | ERP Action |
| --- | --- | --- | --- |
| `anomaly-select-alert` | `anomaly-alerts` | Alert object | Open or mount `anomaly-alert-detail`. |
| `anomaly-status-change` | `anomaly-alert-detail` | Feedback response | Refresh the alert list. |
| `anomaly-close` | `anomaly-alert-detail` | none | Hide detail panel. |
| `anomaly-score-received` | `anomaly-score` | Invoice score result | Show warning or block invoice flow. |

## Styling

The SaaS configurator stores widget configuration in local storage under:

```txt
widgetsapp.saas.widgetData.v1
```

The Ask&Go ERP page reads that configuration and applies these CSS variables:

```css
--anomaly-primary-color
--anomaly-primary-dark
--anomaly-accent-color
--anomaly-background
--anomaly-surface
--anomaly-surface-hover
--anomaly-text-primary
--anomaly-text-secondary
--anomaly-text-muted
--anomaly-border-color
--anomaly-border-radius
--anomaly-danger
--anomaly-success
--anomaly-warning
--anomaly-info
--anomaly-font-family
--anomaly-font-mono
```

When shared style fields are changed for an Ask&Go widget, the frontend syncs those fields to all Ask&Go widgets so `/askgo` stays visually consistent.

## Development Notes

- Backend data is in memory. Restarting `backend/server.js` resets alerts, invoices, suppliers, and departments.
- The backend is intentionally permissive and uses `cors()` for local development.
- The mock SSO token is not validated server-side.
- `frontend/dist/` is generated by `npm run build` and should not be edited manually.
